📋 Resumen ejecutivo

El MVP de TodasMisCosas está bien construido para su etapa: el flujo completo reserva → pago → escrow → chat → acceso → liberación fue validado en producción el 11 y 12 de junio de 2026 sin errores. El stack es sólido (Next.js 14 + Express + MySQL + Supabase Auth + MercadoPago), los emails funcionan, los cron jobs están activos y el deploy vía GitHub Actions es confiable.

Este documento identifica 26 oportunidades de mejora ordenadas por impacto — desde gaps de seguridad puntuales hasta mejoras de arquitectura que serán necesarias en la etapa de escala. Ninguno de los ítems rompe el producto hoy; todos son inversiones para el crecimiento.

Seguridad
Gaps puntuales — ninguno compromete el MVP pero deben resolverse antes de escalar
Crítico
API Key de Google Maps expuesta al navegador
NEXT_PUBLIC_GOOGLE_MAPS_KEY es visible en el bundle del cliente. Si la key no tiene restricciones de dominio configuradas en Google Cloud Console, cualquiera puede usarla y generar costos inesperados.
Fix: En Google Cloud Console → "Restricciones de aplicación" → HTTP referrers → *.todasmiscosas.com/*. También limitar APIs habilitadas solo a Maps JS API.
Crítico
Sin protección CSRF explícita
El middleware de Express no incluye protección CSRF. Las rutas de escritura (POST/PUT/DELETE) que aceptan cookies de sesión son vulnerables a ataques CSRF desde otros dominios.
Fix: Agregar csurf o implementar el patrón Double Submit Cookie. Helmet está configurado pero no cubre CSRF por defecto.
Alto
Sin Content Security Policy estricta
Helmet agrega headers de seguridad base pero no configura un CSP estricto. Sin CSP, inyecciones XSS pueden cargar scripts externos. El frontend usa Google Maps y Supabase — la lista de dominios permitidos es acotada y manejable.
Fix: Configurar helmet.contentSecurityPolicy() con directivas explícitas en app.js. Prioridad media ya que no hay inputs de HTML libre.

Performance
Mejoras que impactan directamente en velocidad percibida y escalabilidad
Alto
Auth middleware: 2 requests externos por API call
Cada endpoint autenticado llama a supabase.auth.getUser() + una query MySQL a usuarios. Sin caché de token. A 50 usuarios concurrentes eso son 100 requests externos por tick.
Fix: Cachear el resultado en memoria (Map + TTL de 60s) usando el JWT como key. Reduce latencia ~40-80ms por request autenticado.
Alto
Listado de espacios sin paginación real
espaciosController.js:138 tiene un LIMIT 20 hardcodeado. No hay parámetros page ni offset. A medida que crezca el catálogo, usuarios no podrán ver más de 20 espacios.
Fix: Agregar ?page=1&limit=20 al endpoint. Estimar 1-2h de implementación en controller + frontend.
Medio
Migraciones ALTER TABLE en cada arranque de PM2
server.js ejecuta docenas de ALTER TABLE IF NOT EXISTS en cada reinicio. En producción estas operaciones son no-ops, pero agregan ~1-2s de boot y generan queries innecesarias en MySQL.
Fix: Mover a un archivo de migración versionado que Guille corre una sola vez. El runner de migraciones ya existe en backend/src/db/migrate.js.
Medio
panel/page.tsx: 1.895 líneas, un solo componente
El panel completo del usuario es un único componente cliente de 1895 líneas que carga todas las tabs, hooks y estados al mismo tiempo. La carga inicial es pesada aunque el usuario solo vea la tab activa.
Fix: Lazy load de tabs con next/dynamic. Cada tab (calendario, movimientos, mis espacios, chat) puede ser un chunk separado.
Medio
Sin SSR en páginas de listado (SEO bloqueado)
El listado de espacios es 'use client' puro — Google indexa una página vacía. Para un marketplace, el SEO de las fichas de espacio es crítico para la adquisición orgánica.
Fix: Convertir /espacios y cada ficha /espacio/[id] a Server Components con generateMetadata() para título, descripción y OG tags.
Bajo
Health check devuelve versión hardcodeada "1.0.0"
app.js responde version: "1.0.0" en el health check sin importar la versión real del package.json. Hace inútil el monitoreo de versión en producción.
Fix: const { version } = require('../package.json'); res.json({ status: 'ok', version }) — cambio de 1 línea.
Bajo
adminController: initTables() en cada require del módulo
adminController.js llama a initTables() al momento de cargar el módulo, creando 4 tablas con CREATE TABLE IF NOT EXISTS en cada arranque. Menor overhead pero poco elegante.
Fix: Mover al mismo sistema de migraciones del punto anterior. Una sola ejecución, no en runtime.

Arquitectura & Calidad de código
Deuda técnica acumulada — manejable hoy, prioritaria antes de sumar features
Alto
Cero tests automatizados en todo el proyecto
No existe ningún archivo de test (.test.js, .spec.ts) ni en frontend ni en backend. El único chequeo de CI es TypeScript. El flujo de escrow, el ledger y los cron jobs corren sin ninguna red de seguridad.
Fix: Al menos 5 tests de integración para el flujo crítico: reserva → pago → confirmarAcceso → liberación. Framework: vitest + supertest para el backend.
Alto
emailService.js: 997 líneas, templates HTML inline
Todos los templates de email (27 tipos) están escritos como string literals dentro de un mismo archivo de 997 líneas. Cada cambio de diseño requiere editar código. No hay preview fácil sin enviar un mail real.
Fix: Mover templates a archivos .html en services/templates/ cargados con fs.readFileSync en boot. Permite editar HTML sin tocar JS.
Medio
UsuarioTipo TypeScript: falta 'oferente'
types/index.ts define UsuarioTipo = 'usuario' | 'admin'. El tipo 'oferente' nunca está en el union, aunque el sistema distingue claramente entre clientes y proveedores en la DB y en la lógica.
Fix: Agregar 'oferente' al union type y remplazar los checks typeof ad-hoc. Ayuda al compilador a detectar ramas no cubiertas.
Medio
SERVICIOS_ADICIONALES: precios todos en cero
types/index.ts define los servicios adicionales (seguro, embalaje, transporte) con precio: 0. En el panel admin se muestran y se pueden asignar, pero nunca se cobran porque el precio no se resuelve en ningún cálculo de checkout.
Fix: Definir los precios reales o cargarlos desde la DB (tabla servicios_adicionales_config). Completar el cálculo en reservasController.js.
Medio
BACKLOG.md referenciado en CLAUDE.md pero no existe
CLAUDE.md instruye al agente a leer docs/BACKLOG.md antes de tocar código, pero el archivo no existe en el repositorio. Cualquier agente que siga esa instrucción fallará silenciosamente.
Fix: Crear docs/BACKLOG.md con los ítems pendientes actuales, o actualizar CLAUDE.md para quitar la referencia.
Medio
'use client' declarado dos veces en panel/page.tsx
Las líneas 1 y 3 de panel/page.tsx repiten 'use client';. No genera error (Next.js lo ignora) pero indica que el archivo fue editado sin orden en varias sesiones y es una señal de deuda de mantenibilidad.
Fix: Eliminar el duplicado. 30 segundos de trabajo.
Bajo
Twilio configurado en modo "demo" — nunca activado
twilioService.js implementa SMS y WhatsApp pero la env var TWILIO_ACCOUNT_SID tiene el valor placeholder "TWILIO_PENDIENTE". Todos los envíos se loguean en consola sin llegar al usuario.
Fix: Activar con credenciales reales de Twilio, o documentar explícitamente que esta feature está en beta. Es un canal valioso para recordatorios de acceso.
Bajo
MercadoPago siempre usa ARS aunque hay 19 monedas
mercadopagoService.js hardcodea currency_id: 'ARS' en todas las preferencias. Los espacios tienen un campo moneda y el tipo MONEDAS define 19 divisas — el dato está, pero se ignora al cobrar.
Fix: Pasar la moneda del espacio al crearPreferencia() y usarla en la preferencia de MP. Habilita espacios en USD para clientes internacionales.

DevOps e Infraestructura
Riesgos operacionales — se vuelven críticos con más usuarios
Alto
Sin ambiente de staging — todo va directo a producción
Cada push a master impacta inmediatamente en todasmiscosas.com. No existe un ambiente de prueba con datos reales o similares. Un bug en auth o escrow afecta usuarios reales en segundos.
Fix: Branch staging + subdominio staging.todasmiscosas.com con su propia DB. Con el self-hosted runner esto es relativamente sencillo.
Alto
Sin backup automático de la base de datos
El plan de backup documentado cubre snapshots de markdown y documentación, pero no hay un mysqldump automatizado de la DB de producción. Si el VPS falla, se pierden todos los datos de reservas, usuarios y ledger.
Fix: Cron diario en el VPS: mysqldump → compresión → upload a Supabase Storage o S3. Retención de 30 días. Costo: prácticamente cero.
Medio
Runner de CI/CD: punto único de falla
El self-hosted runner corre en el mismo VPS que la app. Si el VPS está caído, el deploy falla Y la app está caída — imposible hacer un deploy de emergencia para restaurar el servicio.
Fix: Agregar un segundo runner en un VPS secundario barato, o migrar el deploy job a un runner de GitHub (ubuntu-latest) con SSH + rsync al VPS.
Medio
Deploy sin rollback: git reset --hard a master
El workflow hace git reset --hard origin/master. Si el build falla a mitad, la app queda en estado inconsistente. No hay blue-green ni la posibilidad de volver al commit anterior sin intervención manual de Guille.
Fix: Agregar un paso de smoke test post-deploy (curl /api/health) con rollback automático al commit previo si falla. ~15 líneas en el YAML.
Bajo
Sin CDN para assets estáticos
Imágenes, JS y CSS del frontend se sirven directamente desde el VPS en Buenos Aires. Usuarios desde el interior o el exterior experimentan mayor latencia. Las fotos de espacios ya van a Supabase CDN; los assets de la app, no.
Fix: Cloudflare gratuito como proxy. Agrega CDN, caché de assets, protección DDoS y HTTPS automático sin cambiar el servidor de origen.

Producto & UX
Gaps que afectan la experiencia del usuario final y la confianza en la plataforma
Alto
Auto-release escrow: el cliente no recibe email
El cron de escrow.js libera automáticamente el pago a las 48h si el cliente no confirmó el acceso. Notifica al admin y al proveedor, pero el cliente no sabe que el escrow se liberó de forma automática — puede generar reclamos.
Fix: Agregar sendAccesoConfirmadoDemandante() en el bloque de auto-release del cron. Email ya existe — solo falta conectarlo. Cambio de 3 líneas en escrow.js.
Alto
Historial de chat inaccesible tras liberar escrow
Cuando se libera el escrow, el chat desaparece para ambas partes (fix de v1.11.1). Correcto como regla de negocio, pero el historial previo se pierde permanentemente. Usuarios no pueden revisar acuerdos escritos en el chat.
Fix: Ofrecer descarga del historial en PDF al momento del cierre, o mostrar el chat en modo "solo lectura" (sin input) después del cierre. Ambas opciones son viables con el schema actual.
Medio
Consultas de contacto sin loop de respuesta por email
Cuando alguien envía un formulario de contacto al admin, el admin puede responder desde el panel pero esa respuesta nunca se envía por email a quien hizo la consulta. El flujo está roto a medias.
Fix: En adminController.responderConsulta(), después del UPDATE, enviar email con la respuesta al email del row de la consulta. Requiere un template nuevo en emailService.
Medio
Sin error boundaries en el árbol de componentes
El frontend no tiene ningún ErrorBoundary. Un error de render en cualquier componente del panel — por ejemplo, un dato inesperado de la API — desmonta toda la pantalla y muestra una página en blanco sin mensaje útil al usuario.
Fix: Agregar error.tsx en las rutas de App Router de Next.js. Cubre el caso automáticamente sin necesidad de componentes ad-hoc.
Medio
Sin loading skeletons en el panel (data jumps)
El panel carga datos asincrónicamente (reservas, espacios, movimientos, chat). Durante la carga, la UI muestra secciones vacías que luego se rellenan bruscamente. Afecta la percepción de velocidad y profesionalismo.
Fix: Skeletons CSS para las secciones principales. Compatibles con el diseño dark actual. Next.js 14 tiene loading.tsx que simplifica esto.
Medio
Twilio SMS/WhatsApp sin activar — canal de alto valor
La infraestructura de SMS y WhatsApp está lista en authController.js (ya dispara en el flujo de auth). Activarla con credenciales reales de Twilio desbloquea recordatorios de fecha de ingreso, confirmación de PIN y alertas de liberación de escrow por WhatsApp — canal preferido en Argentina.
Fix: Contratar plan Twilio básico (~$15/mes), configurar las 3 env vars en el VPS y cambiar TWILIO_PENDIENTE por el SID real. 30 minutos de trabajo de Guille.
Bajo
Foto de perfil: ningún usuario puede cambiarla desde el panel
El componente Avatar muestra la foto de perfil del usuario, pero no existe ningún flujo para subir o cambiar esa foto desde el panel. Los avatares aparecen como iniciales o imagen por defecto permanentemente.
Fix: Un input file en la tab "Mi perfil" del panel, con upload a Supabase Storage bucket avatars/. El pattern de upload ya existe en subirFotos.
Bajo
Sin página de FAQ / ayuda para usuarios
No existe una sección de preguntas frecuentes pública. Las dudas sobre escrow, PIN, cancelaciones y comisiones llegan al formulario de contacto, generando trabajo manual de soporte que podría evitarse.
Fix: Página estática /es/ayuda con acordeón de FAQs. Contenido gestionable por Alejandro, zero backend. Alta palanca/esfuerzo.
✅
Lo que está muy bien hecho
Decisiones de arquitectura que demuestran madurez — mantener en v2.0
📒 Ledger de doble entrada
El sistema contable en movimientos_ledger registra cada peso que entra y sale con debito/crédito balanceado. Nivel bancario. Invaluable para auditorías y futuros reportes fiscales.
🎛️ Feature flags de email
La tabla email_config permite habilitar/deshabilitar cualquier tipo de email sin redeploy. Con caché de 1 minuto. Diseño elegante que evita reinicios por configuración.
🛡️ Filtro de contenido bicapa
El contactFilter corre en frontend (UX) y en backend (seguridad). 4 categorías. Protege contra contacto directo, lenguaje inapropiado y desvío de transacciones.
🗑️ Soft delete con trazabilidad
Los espacios eliminados quedan con activo = FALSE + eliminado_por_oferente. Las reservas históricas siguen siendo consultables. Cero pérdida de datos.
🔤 Charset collision prevention
La regla de nunca hacer JOIN entre consultas_espacio y espacios (collations distintas) está documentada en CLAUDE.md y el código la cumple con dos queries + merge en JS.
🔐 Sistema PIN + QR
El PIN de acceso al espacio se genera al confirmar la reserva, se muestra en el email y el QR está listo en el panel. Flujo de entrega de llaves digital bien pensado.
⏱️ Rate limiting multicapa
200 req/min general, 10/15min en auth, 5/hr en contacto. Bien granular. Evita tanto abuso de API como fuerza bruta en login sin afectar UX normal.
📸 Fotos en Supabase Storage
No hay dependencia de filesystem local del VPS. Las fotos sobreviven redeploys, son servidas por CDN de Supabase y el patrón multer memoryStorage es correcto para este deploy.
🤖 CI TypeScript check
El primer job del workflow valida TypeScript antes de desplegar. Impide que errores de tipos rompan el build de producción. Simple y efectivo.
