Las cinco capas — estado actual
L1
Memory Layer
CLAUDE.md — La Memoria
La constitución del repo. Reglas de nombres, estructura, expectativas. Un archivo global para todos los proyectos y uno local por repo.
⚠ Parcial
Lo que ADK propone
Un CLAUDE.md global en ~/.claude/ para todos los proyectos
Un CLAUDE.md por repo con reglas específicas del proyecto
Archivo architecture.rules separado con convenciones de código
Un sistema de memoria persistente para contexto entre sesiones
Tu estado actual
CLAUDE.md completo en TMC: stack, roles, arquitectura, reglas de código, plugins, deploy
CLAUDE.md iniciado en TME con config básica
Sistema de memoria persistente en ~/.claude/projects/ con 30+ archivos activos
Falta ~/.claude/CLAUDE.md global que aplique a TODOS los proyectos sin repetir reglas en cada repo
L2
Knowledge Layer
Skills — El Conocimiento
Workflows reutilizables que Claude auto-invoca al detectar coincidencia con la descripción de la tarea. Sin slash commands — simplemente sabe.
⚠ Parcial
Lo que ADK propone
Archivos .md en skills/ o .claude/agents/ que Claude invoca por similitud semántica
Scripts de referencia y plantillas en scripts/
Skills compartidas entre todo el equipo (distribuidas vía plugins)
Skills project-specific para flujos únicos de cada repo
Tu estado actual
7 plugins instalados = 55 skills activas (engineering, marketing, sales, product-management, legal, desktop-commander, productivity)
Skills de alcance global disponibles en cualquier sesión
Faltan skills específicas de TMC/TME: cómo deployar TMC, cómo correr una migración, cómo revisar un PR del stack propio
No existe carpeta .claude/agents/ en el repo con workflows propios
L3
Guardrail Layer · 🔴 La mayor brecha
Hooks — Los Guardianes
Shell scripts que corren antes y después de cada tool call. Bloquean comandos peligrosos, auto-lintean al guardar, pingean Slack en deploy. Determinísticos. No AI. No dependen de que Claude lo recuerde.
✗ No existe
Lo que ADK propone
PreToolUse.sh — valida y bloquea antes de ejecutar cualquier herramienta
PostToolUse.sh — auto-lint, auto-format, notificaciones después de cada acción
SessionStart.sh — contexto automático al arrancar: estado del repo, pendientes, salud del sistema
Son shell puro — predecibles, auditables, sin costo de tokens
Lo que te haría falta a vos
Nada de esto existe hoy en el entorno
PreToolUse: bloquear rm -rf en rutas de prod, alertar al tocar emailService.js o controllers de MercadoPago, bloquear force-push a master
PostToolUse: tsc --noEmit automático después de editar cualquier .ts/.tsx de TMC — los errores de TypeScript ya no llegan al push
SessionStart: estado de git TMC/TME, verificar que servidor.py corre en 8765, mostrar migraciones pendientes desde CLAUDE.md
L4
Delegation Layer
Subagents — La Delegación
Agentes aislados con su propio contexto. Un code reviewer que solo ve el diff. Un test runner con permisos propios. Mantiene la sesión principal limpia.
⚠ Básico
Lo que ADK propone
Archivos .md en subagents/ que definen agentes especializados con permisos y contexto propios
code-reviewer.md — solo ve el diff, solo devuelve hallazgos
test-runner.md — puede correr tests, no puede modificar código
explorer.md — solo lectura, para explorar el codebase sin contaminar la sesión principal
Tu estado actual
Usás el Agent tool para delegar tareas (Explore, code-reviewer built-in)
La sesión principal se mantiene razonablemente limpia
Sin definiciones formales de subagentes con permisos específicos
El valor real escala con el equipo — para un setup de 1-2 personas, el Agent tool actual es suficiente
L5
Distribution Layer
Plugins — La Distribución
Empaqueta todo el sistema en una instalación. Cada integrante del equipo recibe los mismos skills, hooks y agentes. Alineados desde el primer día.
✓ Completo
Lo que ADK propone
manifest.json — lista de skills, agents, hooks y comandos a instalar
marketplace.url — paquetes npm para capacidades de agentes
team.install — un comando, todo el equipo alineado
Tu estado actual
7 plugins instalados: engineering · marketing · sales · product-management · legal · desktop-commander · productivity
55 skills activas · ~3,326 tok siempre activos
14 plugins pendientes identificados y priorizados en memoria
No existe un team.install para que Guille replique el mismo setup con un comando
3 prioridades de implementación
🔴 Prioridad 1 · Mayor valor
Hooks — Los Guardianes
Shell scripts determinísticos que corren en cada sesión. No dependen de la memoria de Claude. Previenen errores en producción antes de que ocurran. El mayor ROI de las 5 capas.
PreToolUse.sh — bloquear rm -rf en prod, alertar archivos críticos
PostToolUse.sh — tsc --noEmit automático al guardar .ts
SessionStart.sh — estado git + servidor.py + migraciones pendientes
🟡 Prioridad 2 · Medio plazo
CLAUDE.md global
Un archivo en ~/.claude/CLAUDE.md que aplica a todos los proyectos sin repetir reglas en cada repo. Las convenciones cross-proyecto dejan de vivir solo en memoria.
Docs con tabs → siempre HTML en repo, nunca artifacts
Links siempre localhost o htmlpreview, nunca file://
Español rioplatense, sin emojis salvo pedido explícito
🟡 Prioridad 3 · Cuando Guille use Claude
team.install para Guille
Un script o archivo de config que Guille ejecuta una vez y queda con el mismo entorno: mismos plugins, mismo CLAUDE.md, mismos hooks. Alineados desde el primer prompt.
Instalar los 7 plugins del marketplace en su sesión
Copiar CLAUDE.md de TMC/TME a su entorno global
Configurar hooks básicos (PreToolUse de seguridad)
