🎯 El problema que resuelve

Hoy organizar un evento privado y cobrar por él requiere 4 herramientas distintas que no hablan entre sí.

El perfil de host que nadie sirve bien hoy
Alguien que organiza una cata de vinos para 15 personas, un taller de cerámica en su casa, una clase de yoga privada, una cena club, una fiesta temática o una masterclass informal — y quiere cobrar $X por persona, manejar la lista y no usar Excel + MercadoPago + WhatsApp + Instagram por separado.

Su flujo actual: anuncia en Instagram → gestiona lista por DM → manda link de MP → controla quién pagó en Excel → el día del evento no sabe quién está confirmado.
🔐 Los 4 niveles de privacidad — El diferenciador real

Ninguna plataforma en Argentina combina estos 4 niveles con cobro integrado y gestión de acceso.

1
Público
Visible en el feed de descubrimiento. Cualquier usuario puede verlo y comprar su ticket. Ideal para talleres, clases, meetups, eventos culturales.
2
Solo con link
No aparece en el feed. Accesible únicamente con la URL directa que el host comparte. Cualquiera con el link puede comprar.
3
Privado con solicitud
El interesado solicita asistir. El host aprueba o rechaza cada solicitud antes de habilitar el pago. Control total sobre quién entra.
4
Lista cerrada
Solo pueden entrar los invitados cargados explícitamente por el host con nombre y email. Máxima exclusividad — nadie que no esté en la lista puede comprar.

⚠️ El riesgo más grande — Desintermediación

El host consigue asistentes en la plataforma y los mueve a WhatsApp. El clásico problema del marketplace de servicios.

🚨
Por qué puede pasar
Una vez que el host tiene el WhatsApp de sus asistentes puede organizar el próximo evento por fuera, evitar la comisión y manejar el pago con un link de MP directo.
🛡️
Por qué es difícil acá
1. El QR lo genera la plataforma — sin la app no puede controlar el acceso el día del evento.
2. La reputación no se porta — sus reseñas y historial están acá. Empezar de cero es costoso.
3. El descubrimiento es de la plataforma — los asistentes nuevos (no amigos) lo encuentran en el feed.
4. El escrow genera confianza — un link de MP no protege al asistente si el host cancela.

🔄 TMC vs Eventos — La diferencia estructural
🏠
TMC — Escrow 1 a 1
1 cliente → 1 pago → 1 proveedor
Trigger: acción humana (confirmar acceso)
Duración: semanas o meses
Liberación: un solo pago
🎟️
Eventos — Escrow 1 a N
N asistentes → N pagos → 1 host
Trigger: evento completado (automático + manual)
Duración: horas o días
Liberación: todos los pagos simultáneos
💫 Los 3 momentos donde aplica el escrow
💳
Momento 1 — Compra del ticket
El asistente paga. El dinero queda retenido en MercadoPago. No va al host todavía. El sistema genera el ticket con QR único y lo manda por email.
-- En este momento se define el split Asistente paga: $10.000 └── Plataforma (8%): $800 ← retenido └── Host (92%): $9.200 ← retenido hasta trigger ticket.escrow_estado = 'retenido' ticket.qr_token = uuid() único
📱
Momento 2 — Check-in el día del evento
El host escanea el QR. El sistema registra la asistencia pero el dinero sigue retenido. El check-in no libera — solo confirma que el asistente llegó.
Host escanea QR └── check_ins.verificado = TRUE └── ticket.asistio = TRUE └── ticket.checkin_at = NOW() └── dinero = sigue retenido ← importante
🔓
Momento 3 — Cierre del evento (trigger de liberación)
Acá se libera todo. Dos mecanismos combinados: el host cierra manualmente o el cron job lo hace automáticamente 3 horas después.
-- OPCIÓN A: Manual POST /eventos/:id/cerrar └── escrowService.liberarEvento(eventoId) └── Para cada ticket WHERE escrow_estado = 'retenido': └── MP API: release payment └── ticket.escrow_estado = 'liberado' └── ticket.liberado_at = NOW() -- OPCIÓN B: Automático (fallback si el host olvidó cerrar) escrowRelease.js (cron cada 30 min) └── IF evento.fecha_fin < NOW() AND estado != 'cancelado' └── liberar automáticamente a las 3hs post-evento
⚠️ Escenarios especiales
🚫 Host cancela el evento
Host presiona "Cancelar evento" → escrowService.cancelarEvento(eventoId) → Para cada ticket con escrow_estado = 'retenido': MP API refund → ticket.escrow_estado = 'reembolsado' → Email al asistente: "El evento fue cancelado. Tu dinero fue devuelto en 3-5 días hábiles."
👻 Host no aparece — posible fraude
Cron detecta que el evento terminó hace +3hs y el estado sigue activo. NO libera automáticamente — abre disputa: evento.estado = 'en_disputa'. Revisión manual del equipo TMC → reembolso total a todos los asistentes. Este caso es la diferencia clave con el cierre normal.
🔙 Asistente cancela su ticket
Política configurable por el host al crear el evento: > 48hs antes → reembolso 100% · 24-48hs → reembolso 50% · < 24hs → sin reembolso (el host ya incurrió en costos de organización). El sistema calcula automáticamente el porcentaje.
🪑 Asistente pagó pero no apareció (no-show)
ticket.asistio = FALSE al cierre del evento → dinero se libera igual al host. Compró el derecho a asistir, no la asistencia en sí. Igual que una aerolínea con un no-show. Sin reembolso automático — si quiere reclamar, debe abrir una disputa.

CREATE TABLE tickets ( id INT AUTO_INCREMENT PRIMARY KEY, evento_id INT NOT NULL, usuario_id INT NOT NULL, mp_payment_id VARCHAR(100), -- ID del pago en MercadoPago mp_preference_id VARCHAR(100), precio DECIMAL(10,2), -- precio total pagado comision DECIMAL(10,2), -- 8% calculado al momento de compra neto_host DECIMAL(10,2), -- 92% a transferir al host escrow_estado ENUM( 'pendiente', -- esperando confirmación de MP 'retenido', -- pago confirmado, dinero en escrow 'liberado', -- dinero transferido al host 'reembolsado' -- devuelto al asistente ) DEFAULT 'pendiente', qr_token VARCHAR(255) UNIQUE, -- genera el QR del ticket asistio BOOLEAN DEFAULT FALSE, checkin_at DATETIME, liberado_at DATETIME, created_at DATETIME DEFAULT NOW() );

⚠️ El riesgo operativo a vigilar — RAM del VPS
🔥
El único riesgo compartido con TMC es el VPS. Si el proceso de Eventos consume demasiada RAM y el servidor se queda sin memoria, PM2 puede afectar a TMC.
