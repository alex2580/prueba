1. Resumen Ejecutivo

TodasMisCosas.com es un marketplace de almacenamiento urbano en Buenos Aires que conecta a personas que tienen espacio disponible (oferentes) con personas que necesitan guardar cosas (demandantes).

Qué haceCómo lo hace
Publicar y buscar espacios de almacenamientoMapa interactivo + filtros por barrio, tipo, precio
Reservar un espacio por días o mesesFlujo de 3 pasos + cálculo automático de precio
Cobrar y pagar de forma seguraIntegración con MercadoPago (Checkout Pro + QR)
Comunicar oferente y demandanteChat en tiempo real vía WebSocket (Socket.io)
Autenticar con doble factorEmail/contraseña (Supabase) + OTP de 6 dígitos
Notificar eventos claveEmails automáticos (Nodemailer + Hostinger SMTP)
Gestionar el negocioPanel de admin con estadísticas, gestión de usuarios y espacios

Roles de usuario

RolQué puede hacer
DemandanteBuscar espacios, reservar, pagar, chatear, dejar reviews
OferentePublicar espacios, ver y gestionar reservas recibidas, chatear
AdminTodo lo anterior + gestionar usuarios, espacios, reservas, ver estadísticas globales

8. Flujo de Seguridad — Autenticación 2FA / OTP

Visión general

La autenticación usa dos factores: primero email + contraseña vía Supabase, luego un código OTP de 6 dígitos enviado al usuario.

Flujo completo de login

[1] Usuario ingresa email + contraseña en LoginForm ↓ [2] Frontend llama a Supabase signIn(email, password) → Supabase verifica credenciales → Devuelve JWT (session.access_token) ↓ [3] Frontend llama a POST /api/auth/solicitar-otp → Header: Authorization: Bearer <token> → Backend: middleware requireAuth valida el JWT → Backend: invalida OTPs anteriores del usuario → Backend: genera código de 6 dígitos (100000-999999) → Backend: guarda en tabla auth_otp (expira en 10 min, max 3 intentos) → Backend: envía el código por: • Email (Nodemailer + Hostinger SMTP) — SIEMPRE ✅ • SMS (Twilio) — solo si el usuario tiene teléfono cargado • WhatsApp (Twilio) — solo si el usuario tiene teléfono cargado → Respuesta: { ok: true, canales: {...}, email_hint: "al***@gmail.com" } ↓ [4] Frontend muestra pantalla OTPStep (6 inputs de 1 dígito c/u) ↓ [5] Usuario ingresa código → auto-submit cuando están los 6 dígitos ↓ [6] Frontend llama a POST /api/auth/verificar-otp → Header: Authorization: Bearer <token> → Body: { codigo: "123456" } → Backend: busca OTP activo más reciente del usuario → Si expiró: error "El código expiró" → Si mal código: incrementa intentos (máx 3), devuelve cuántos quedan → Si 3 intentos fallidos: invalida el OTP, pide nuevo → Si correcto: marca OTP como usado, registra sesión en auth_sesiones ↓ [7] Backend: envía notificación de acceso exitoso al usuario (email + SMS/WhatsApp con fecha, IP y dispositivo) ↓ [8] Frontend: llama a GET /api/usuarios/me con el token → Carga el perfil completo del usuario → Setea user + token en el estado global (useAuth) ↓ [9] Usuario autenticado — acceso completo a la plataforma

Middleware de autenticación (requireAuth)

Cada request protegida pasa por backend/src/middleware/auth.js:

Recibe request con header: Authorization: Bearer <token> ↓ Llama a supabase.auth.getUser(token) ↓ Si error o token inválido → 401 "Token inválido o expirado" ↓ Busca usuario en MySQL por supabase_id → Si no existe por supabase_id → busca por email y linkea → Si no existe en ningún lado → lo crea automáticamente ↓ Si usuario.activo = false → 403 "Cuenta suspendida" ↓ Adjunta req.user = { id, nombre, email, tel, tipo, supabase_id } ↓ Llama a next() → pasa al controller

Cambio de teléfono con OTP

Para cambiar el número de teléfono se usa un flujo OTP separado con tipo = 'cambio_tel':

[1] Usuario ingresa nuevo número en el modal de perfil [2] POST /api/usuarios/me/solicitar-cambio-tel { tel_nuevo } → Genera OTP en auth_otp con tipo='cambio_tel', tel_nuevo=<número> → Envía código al NUEVO número por SMS + WhatsApp [3] Usuario ingresa el código de 6 dígitos [4] POST /api/usuarios/me/verificar-cambio-tel { codigo } → Valida OTP con tipo='cambio_tel' → Actualiza usuarios.tel = tel_nuevo → Envía email de confirmación "Tu teléfono fue actualizado"

Bloqueo de cuentas

Los administradores pueden desactivar cuentas desde el panel de admin. Si usuarios.activo = false, el middleware devuelve 403 en cada request. El usuario no puede entrar hasta que un admin lo reactive.

Capas de seguridad HTTP

CapaQué protege

9. Flujo de Pagos — MercadoPago

Modalidades disponibles

ModalidadDescripciónCómo funciona
💳 Pago onlineCheckout Pro de MercadoPagoRedirige al usuario a la página de pago de MP
📱 Pago por QRQR generado en pantallaMuestra QR, el usuario escanea con la app MP

Flujo completo de pago online

[1] Usuario en /espacio/:id/reservar — Paso 3 (Cuenta y Pago) Usuario logueado y verificado ✅ ↓ [2] Click en "💳 Pagar online con MercadoPago" ↓ [3] Frontend: POST /api/reservas Body: { espacio_id, fecha_desde, fecha_hasta } → Backend verifica: espacio existe, está activo y disponible → Backend verifica: no hay overlap con otras reservas → Backend calcula precio: • Si < 28 días: días × precio_dia • Si ≥ 28 días: ceil(días/30) × precio_mes → Backend crea registro en tabla reservas (estado: "pendiente") → Backend envía emails de notificación (ver Sección 10) → Respuesta: { id: "uuid-de-reserva", precio_total: 18000, ... } ↓ [4] Frontend: POST /api/pagos/preferencia Body: { reserva_id: "uuid-de-reserva" } → Backend crea preferencia en MercadoPago: • título: "Reserva Cochera Palermo — Palermo" • monto: precio_total • back_urls: success/failure/pending → /reserva/:id/confirmacion • external_reference: reserva_id • notification_url: /api/pagos/webhook → MP devuelve init_point (URL de pago) + preference_id → Backend guarda mp_preference_id en la reserva → Respuesta: { init_point: "https://www.mercadopago.com.ar/..." } ↓ [5] Frontend: window.location.href = init_point → Usuario va a la página de pago de MercadoPago ↓ [6] Usuario paga en MercadoPago (tarjeta, débito, dinero MP, etc.) ↓ [7] MercadoPago llama al webhook: POST /api/pagos/webhook → Backend obtiene el pago por su ID → Si approved → reserva.estado = "pagada", guarda mp_payment_id → Si rejected/cancelled → reserva.estado = "cancelada" → Backend envía emails de confirmación de pago (ver Sección 10) ↓ [8] MercadoPago redirige al usuario: • Si aprobado → /reserva/:id/confirmacion?estado=success • Si rechazado → /reserva/:id/confirmacion?estado=failure • Si pendiente → /reserva/:id/confirmacion?estado=pending ↓ [9] Frontend: página de confirmación → Consulta GET /api/pagos/estado/:reservaId para obtener estado real → Muestra: 🎉 "¡Reserva confirmada!" / ❌ "Pago rechazado" / ⏳ "Pago pendiente"

Flujo de pago por QR

[1–4] Igual que el flujo online (crea reserva + preferencia en MP) ↓ [5] Frontend genera QR a partir del init_point: QRCode.toDataURL(pref.init_point) → imagen PNG en base64 ↓ [6] Se muestra el QR en pantalla + botón "Abrí en otro dispositivo" ↓ [7] Frontend inicia polling cada 4 segundos: GET /api/pagos/estado/:reservaId → Cuando estado === "pagada" → router.push('/reserva/:id/confirmacion') ↓ [8] Usuario escanea con app de MercadoPago → paga ↓ [9] MP dispara webhook → estado cambia a "pagada" en DB [10] Polling detecta estado "pagada" → redirige a confirmación ✅

Extensión de reservas pagadas

Un demandante puede extender la fecha de fin de una reserva ya pagada:

POST /api/reservas/:id/extender { nueva_fecha_hasta } → Verifica que no hay overlap con otras reservas en ese período → Calcula precio adicional (días extra × tarifa) → Crea registro en reserva_extensiones → Crea nueva preferencia MP → Devuelve init_point para pagar la extensión → Webhook actualiza fecha_hasta en la reserva + envía email de confirmación

Cron jobs de recordatorios

Archivo: backend/src/jobs/recordatorios.js — corre todos los días a las 09:00 hs (Argentina).

Busca todas las reservas en estado "pagada" con fecha_hasta próxima: → Con 5 días para vencer y recordatorio_5d = 0 → envía email → recordatorio_5d = 1 → Con 2 días para vencer y recordatorio_2d = 0 → envía email → recordatorio_2d = 1 → Con 1 día para vencer y recordatorio_1d = 0 → envía email → recordatorio_1d = 1 → Que vence hoy y recordatorio_0d = 0 → envía email → recordatorio_0d = 1

Cron job de inactividad

Archivo: backend/src/jobs/inactividad.js — corre todos los días a las 08:00 hs (Argentina).

Busca espacios activos donde COALESCE(ultima_actividad, created_at) < hace 90 días: → Pone activo = FALSE, inactiva_auto = 1 → Envía email al oferente avisando que su publicación fue pausada El oferente puede reactivar desde su panel → POST /api/espacios/:id/reactivar → activo = TRUE, inactiva_auto = 0, ultima_actividad = NOW()

Flujo de deploy automático

Desarrollador hace push a la rama master en GitHub ↓ GitHub Actions detecta el push → dispara el workflow ↓ Job: deploy (ubuntu-latest) → Usa appleboy/ssh-action para conectarse al VPS por SSH → Ejecuta en el VPS: cd /var/www/todasmiscosas git reset --hard HEAD ← descarta cambios locales ./deploy.sh ← script de build y restart ↓ deploy.sh en el VPS hace (aprox): → git pull origin master → cd frontend && npm install && npm run build → cd backend && npm install → pm2 restart tmc-backend ↓ Migraciones de DB (en el mismo workflow): → node src/db/add-disponibilidad.js → node src/db/add-categoria-seguridad.js → node src/db/add-moneda-perfil.js → node src/db/add-recordatorios-extensiones.js → node src/db/add-bloqueo-usuarios.js → node src/db/add-auth-otp.js → node src/db/add-perfil-inactividad.js (cada script es idempotente: ignora si la columna ya existe) ↓ App actualizada en producción ✅ (~2-3 minutos desde el push)

14. Tareas Pendientes

🔴 BLOQUEADOR PRODUCCIÓN — Twilio sin configurar El OTP por SMS y WhatsApp no funciona en producción. El OTP por email sí funciona. Para activar SMS y WhatsApp, hay que agregar estas 4 variables en el .env del VPS:

1. Crear cuenta en twilio.com
2. Console → Account Info → copiar TWILIO_ACCOUNT_SID y TWILIO_AUTH_TOKEN
3. Comprar número de teléfono en Twilio (sección Phone Numbers)
4. Activar Sandbox de WhatsApp: Messaging → Try it out → WhatsApp
5. SSH al VPS → editar /var/www/todasmiscosas/backend/.env → agregar las 4 variables
6. pm2 restart tmc-backend
7. Probar login y verificar que llega el código por SMS y WhatsApp
🟡 Backlog — Historial de cambios de perfil (Backlog D) Falta implementar un log de auditoría que registre cuándo el usuario cambió su nombre, teléfono o dirección. Útil para soporte y para detectar actividad sospechosa. No es bloqueante.
